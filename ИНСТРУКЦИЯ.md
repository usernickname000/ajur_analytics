# Инструкция по работе с аналитикой заказов

Документ описывает как программа считает выручку, откуда берутся данные,
как сверяться с бухгалтерией и что менять при переходе на новый год.

---

## 1. Откуда берутся цифры в отчёте

Программа работает с **двумя источниками данных**:

### Источник 1 — CRM (Битрикс)
Выгрузка заказов `заказы_YYYYг._обновлено_DD.MM.YYYYг.xlsx`. Это основа.
Из неё считаются **все разрезы**: клиенты, менеджеры, отрасли, RFM, номенклатура, сезонность.

### Источник 2 — `external_income.json`
Ручной файл с внешними доходами, которых нет в CRM:
- **Программатик** (ведётся во внешних платформах, ~119 млн/год)
- Бартер, гранты, ИРИ, взаимозачёты (опционально)

Эти данные попадают только в **сводные цифры** (общий итог, по месяцам,
по кварталам, по сезонам). В разрезы клиентов/менеджеров/отраслей они
не добавляются — там нет информации откуда к какому клиенту отнести сумму.

---

## 2. Что показывает лист `01_Месячная_статистика`

В месячной таблице теперь **пять разных сумм выручки**:

| Колонка | Что считает | Когда использовать |
|---|---|---|
| Сумма выручки, тыс. руб. (CRM, дата заказа) | Только CRM, группировка по дате заказа | Внутренний анализ продаж |
| Выручка по дате оплаты, тыс. руб. | Только CRM, но по дате оплаты | **Сверка с бухгалтерией** |
| Внешние доходы, тыс. руб. | Программатик из JSON | — (служебная) |
| Итого с внешними, тыс. руб. | CRM (дата заказа) + программатик | Для презентации — "как было" |
| **Итого по оплате + внешние, тыс. руб.** | CRM (дата оплаты) + программатик | **Для презентации — совпадает с 482 404** |

### Какую колонку использовать в презентации

Для максимального совпадения с верифицированными цифрами бухгалтерии —
**«Итого по оплате + внешние»**. Даёт погрешность около 1%.

---

## 3. Три верифицированные цифры года

Эти константы в начале `analytics.py` должны соответствовать реальности.
На данный момент:

```python
VERIFIED_TOTAL_WITH_PROG           = 482_404_000  # всё с программатиком и бартером
VERIFIED_TOTAL_WITH_BARTER_NO_PROG = 363_000_166  # с бартером, без программатика
VERIFIED_ADVERTISING_NO_EVENTS     = 243_005_820  # рекламная без мероприятий
```

Эти цифры приходят из верифицированного отчёта бухгалтерии. Менять их
нужно вручную раз в год при подготовке годового отчёта.

---

## 4. Как программа считает выручку: три варианта

### Вариант А — как было раньше (по дате заказа)
Группировка по колонке «Месяц» из CRM = месяц, к которому заказ был
отнесён менеджером при создании. Даёт 382 023 тыс. за 2025.

**Плюс:** вся выручка 2025 года попадает в отчёт за 2025.
**Минус:** не бьётся с бухгалтерией на уровне каждого месяца.

### Вариант Б — по дате оплаты (новая колонка)
Группировка по `ДатаСуммаОплаты_`. Даёт 358 723 тыс. за 2025.

**Плюс:** бьётся с бухгалтерией с погрешностью ~1%.
**Минус:** заказы, оплаченные после 31 декабря, уезжают в следующий год
(11 млн в янв.2026, 3 млн в 2024).

### Вариант В — CRM + внешние доходы (то что нужно для презентации)
Вариант Б + данные из `external_income.json`. Даёт 477 491 тыс. за 2025.

**Цель бухгалтерии: 482 404. Разница: 1%.**

## 5. Как работать с `external_income.json`

Файл лежит рядом с `analytics.py`. Формат — JSON с помесячной разбивкой
каждой строки:

```json
"Программатик ФОНТАНКА": {
  "01": 4611923.68, "02": 4943308.41, ... "12": 8767835.58
}
```

**Цифры в рублях, не в тысячах.** Если месяц ещё не закрылся — ставь 0.

### Служебный ключ `_включать_в_аналитику`

Это список строк, которые прибавляются к CRM при расчёте `Внешние доходы`.
Сейчас там только программатик:

```json
"_включать_в_аналитику": [
  "Программатик ФОНТАНКА",
  "Программатик ФОНТАНКА ТГ",
  "Программатик ДОКТОР",
  "Программатик ДОКТОР ТГ"
]
```

Если хочешь добавить, например, бартер в общую картину — допиши в список.
Но имей в виду: бартер это не «реальная выручка», а внутренний зачёт,
поэтому обычно не добавляется.

## 6. Обновление данных раз в год (при переходе на 2026 год)

Когда придёт время делать отчёт за 2026 год:

### Шаг 1. В `external_income.json` поменять год
```json
"_год": 2026,
```

### Шаг 2. Заменить цифры программатика
Взять из бухгалтерской таблицы Доходы_2026.xlsx строку «Программатик ФОНТАНКА»
(и другие программатики) по каждому месяцу — переписать в JSON. В рублях.

### Шаг 3. В `analytics.py` обновить верифицированные константы
```python
VERIFIED_TOTAL_WITH_PROG           = 500_000_000  # цифра за 2026
VERIFIED_TOTAL_WITH_BARTER_NO_PROG = 380_000_000
VERIFIED_ADVERTISING_NO_EVENTS     = 260_000_000
```

### Шаг 4. Проверить маппинг проектов
Если в CRM появились **новые проекты** (например, «Фонтанка.ру: Подкасты») —
добавь их в `PROJECT_GROUP_MAP` в `analytics.py`. Иначе они попадут в
«НЕ КЛАССИФИЦИРОВАНО».

---

## 7. Проверка что всё работает

После запуска программы открой лист `01_Месячная_статистика` и просуммируй
колонку **«Итого по оплате + внешние, тыс. руб.»**. Должно получиться
значение близкое к `VERIFIED_TOTAL_WITH_PROG / 1000` (в пределах 1-2%).

Если расхождение больше 5% — что-то не так:
- проверь что JSON существует и читается (в логе должна быть строка
  «Внешние доходы подгружены: XXX тыс. руб.»)
- проверь что дата оплаты у большинства заказов распознана
- проверь что константы в `analytics.py` соответствуют текущему году

---

## 8. Что НЕ считается с внешними доходами

Эти разрезы считаются **только по CRM** (по причине — во внешних данных
нет клиентов, менеджеров, отраслей):

- `02_Топ_клиентов` — только клиенты из CRM
- `03_Топ_менеджеров` — только из CRM
- `04_Отрасли` — только из CRM
- `04а_Номенклатура` — только из CRM
- `07_RFM_все`, `08_RFM_сегменты`, `09_RFM_не_топ` — только из CRM
- `10_Распределение_заказов` — только из CRM

Поэтому если считать долю топ-клиента от «полной» выручки 482 млн, нужно
делить сумму клиента на **482 млн**, а не на сумму топа клиентов.
Напоминание: этот расчёт делается **руками в презентации**, не в программе.

## 9. Возможные улучшения в интерфейсе программы

Текущая программа запускает аналитику кнопкой. Полезные функции, которые
имеет смысл добавить в GUI:

### A. Переключатель «по дате заказа / по дате оплаты»
Сейчас обе колонки попадают в отчёт. Можно добавить в интерфейс выбор,
какая из них будет «основной» — это поменяет топ-клиентов, топ-менеджеров
и отрасли тоже (сейчас они всегда по дате заказа). Реализуется флагом
в GUI, который передаётся в `run_analytics(..., group_by='payment')`.

### B. Форма для редактирования `external_income.json`
Сейчас пользователь редактирует JSON руками в текстовом редакторе, что
неудобно и опасно (легко сломать синтаксис). Полезно сделать отдельное
окно: таблица 12 месяцев × 15 статей доходов, с кнопкой «Сохранить».

### C. Форма для редактирования верифицированных цифр
Сейчас `VERIFIED_TOTAL_WITH_PROG` и другие — это константы в `.py` файле.
Обычный пользователь их трогать не должен. Перенести в отдельный конфиг
(`verified.json`) и сделать форму редактирования в GUI.

### D. Индикатор расхождения с бухгалтерией
На главном экране после запуска аналитики показывать маленькую сводку:
```
CRM по оплате + программатик: 477 491 тыс.
Цель (бухгалтерия):           482 404 тыс.
Расхождение:                  −1.0% ✅
```
Если расхождение больше 5% — красным. Помогает быстро понять, что
что-то не так с данными.

### E. Сравнение двух периодов
Кнопка «Сравнить с прошлым годом» — выбрать два файла выгрузки и увидеть
таблицу с динамикой по каждому месяцу/сегменту/клиенту. Это заменит
ручное составление презентаций.

### F. Автообновление
Если включён `watcher.py` (уже есть в проекте) — добавить в GUI статус:
«Отслеживается папка X, последнее обновление в 14:32».

## 10. Краткая шпаргалка

**Что поменялось в этой версии:**
- Добавлена колонка «Выручка по дате оплаты» — для сверки с бухгалтерией
- Добавлена колонка «Итого по оплате + внешние» — главная цифра для презентации
- `external_income.json` теперь участвует в месячной/квартальной/сезонной статистике

**Что должно совпадать:**
- Сумма «Итого по оплате + внешние» за год ≈ `VERIFIED_TOTAL_WITH_PROG` (±1-2%)
- Сумма «Выручка по дате оплаты» за год ≈ `VERIFIED_TOTAL_WITH_BARTER_NO_PROG` (±1-2%)

**Что делать при новом годе:**
1. Обновить цифры в `external_income.json` (программатик по месяцам)
2. Обновить три константы `VERIFIED_*` в начале `analytics.py`
3. Проверить что новые проекты CRM есть в `PROJECT_GROUP_MAP`

**Если что-то не бьётся:**
- Проверь лог на наличие «Внешние доходы подгружены: XXX тыс. руб.»
- Открой `01_Месячная_статистика` и сравни колонки
- Погрешность 1-5% — норма (разная методология CRM и бухгалтерии)
- Погрешность 10%+ — что-то сломано, ищи причину
